# s02-update-migration

**Outcome.** Existing gremlins that update past this point gain a populated `context/system/` automatically. Local user content in `context/` is preserved.

## Scope

Modify `commands/update.sh` (or `bin/update.sh`):

- After the canonical overlay completes, invoke `bin/doctor.sh`.
- Doctor's output is folded into `/update`'s final report so the user sees what was added.

`/update` excludes:

- Top-level `context/*.md` remains local state (already the case).
- The `context/system/` directory itself can be allowed to overlay or be skipped — pick the simpler rule and document it. Recommended: skip the directory in overlay and let doctor recreate it; that keeps overlay rules narrow.
- `context/system/README.md` is generated by doctor, so a stale local copy is harmless — doctor will leave it if present.

## Constraints

- Doctor must be idempotent. Running `/update` twice in a row produces the same result and no spurious actions.
- This stitch does **not** add a durable opt-out for system symlinks. A user who `rm`s `context/system/memory.md` will see it restored by the next `/update`. Document this in the README from s01 so the behaviour is at least visible. A real opt-out is a follow-up stitch if friction shows up.

## Verification

1. Older gremlin with no `context/system/` runs `/update` against a canonical tarball. `ls context/system/` after update shows the seeded set.
2. Run `/update` a second time. Doctor reports `ok` for every entry; no actions taken.
3. A user-authored `context/my-notes.md` survives the update unchanged.
4. A user-authored real file `context/system/notes.md` survives the update unchanged (doctor reports `skipped (real file)`).

## Notes

- Calling doctor from `/update` keeps the migration story simple: there is only one place that knows the manifest (doctor), and `/update` just invokes it.
- If the directory overlay-skip rule turns out to be wrong (e.g. canonical needs to ship updated README copy), this can be revisited; doctor is forgiving either way.
